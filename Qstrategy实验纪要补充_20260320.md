# Qstrategy 实验纪要

## 文档目的

本文档用于总结此前围绕 BRICK / 相似度 / 机器学习增强 / 账户层回测 / 规则边界梳理 所做的主要实验，统一记录每项实验的：

- 实验目的
- 一句话结果

---

## 1. baseline 原始策略回测
**目的：** 先确认不加相似度、不加机器学习时，原始规则策略本身的账户层表现。  
**结果：** baseline 作为后续所有实验的基准线，表现不差，但后面被部分相似度方法和 LightGBM 增强版超越。

## 2. 纯相似度筛选初测（corr / euclidean / corr_cosine_pipeline）
**目的：** 验证历史成功形态相似度能不能作为 BRICK 候选池的二次过滤器。  
**结果：** 纯相似度能提胜率，但很多版本总收益明显下降，说明“筛得更准”不等于“账户层更赚钱”。

## 3. corr 高阈值实验
**目的：** 看高相似度阈值是否能筛出更像成功模板的强信号。  
**结果：** 高阈值 corr 往往胜率更高，但覆盖率明显下降，容易变成样本太窄的强筛选器。

## 4. euclidean / corr_cosine 每日 topN 排序实验
**目的：** 不再只做硬阈值过滤，而是每天按相似度分数排序取前 N，看看账户层是否改善。  
**结果：** daily topN 比单纯阈值版稳定得多，尤其 top8、top9 一带表现明显优于很多早期纯过滤方案。

## 5. weighted 分仓版相似度实验
**目的：** 看在相似度策略上增加加权分仓，是否能进一步优化资金曲线。  
**结果：** 有一定改善，但整体仍未明显超过后来的机器学习增强主线。

## 6. top3 / top4 / top5 / top6 / top7 / top8 / top9 / top10 枚举
**目的：** 找到账户层下最合适的每日入选数量。  
**结果：** top8、top9 通常在收益、回撤、覆盖率之间更平衡，过小 topN 容易收益不足，过大 topN 容易稀释优势。

## 7. walk-forward / strict 切分验证
**目的：** 检查相似度增强是否只是测试集反复使用后的假象。  
**结果：** 严格切分后，相似度增强依然有效，但更像“增强器”而不是最终最优完整策略。

## 8. cluster_100 / recent_100 / sample_300 模板构建对比
**目的：** 比较不同成功模板库构建方式对相似度筛选效果的影响。  
**结果：** 聚类代表模板更接近“综合历史成功形态”，但最终是否更优仍要看具体 scorer 和账户层结果。

## 9. close_norm / returns / close_vol_concat 表示法对比
**目的：** 比较价格归一化、收益率序列、价格+成交量拼接哪种更适合做相似度表示。  
**结果：** 单一表示各有优劣，价格+量能拼接更全面，但也更重，最终要结合 scorer 和账户层表现判断。

## 10. corr / cosine / euclidean / dtw / paa / sax / wavelet / fft / pipeline_corr_dtw 覆盖式实验设计
**目的：** 系统比较多种形态相似方法，寻找最适合 BRICK 候选池的相似度方案。  
**结果：** corr、euclidean 更实用，DTW 有价值但开销大，SAX/PAA/小波/FFT 没成为最终主线。

## 11. corr + DTW pipeline 实验
**目的：** 先用相关系数粗筛，再用 DTW 精筛，验证“快筛 + 精筛”是否比单一相似度更有效。  
**结果：** 这条思路合理，但是否优于简单方法要看参数和账户层，工程成本明显更高。

## 12. supervised similarity 实验设计
**目的：** 不只和成功模板比，还同时和失败模板比，再用模型学出一个监督式相似度分数。  
**结果：** 思路上是正确升级方向，但计算量很大，且要防止训练/验证/测试污染。

## 13. LightGBM 相似度增强版初测
**目的：** 验证“相似度 + 结构特征 + 技术特征”交给 LightGBM 排序，是否优于纯相似度。  
**结果：** LightGBM 版本明显强于纯 euclidean / corr 排序，说明相似度单独用不如融合用。

## 14. XGBoost 同框对比
**目的：** 比较 XGBoost 和 LightGBM 在同一特征集上的表现差异。  
**结果：** XGBoost 也强，但整体没有稳定压过最优 LightGBM 版本。

## 15. full 特征版 vs core 特征版 LightGBM
**目的：** 验证精简特征是否比全特征更稳、更强。  
**结果：** core 版通常不弱于 full 版，很多时候还更强，说明减少噪音和冗余是对的。

## 16. core10 / core12 / core15 / core20 特征收缩实验
**目的：** 找到最小但仍然有效的核心特征集。  
**结果：** core10 / core12 一带表现很强，特征更多不一定更好。

## 17. LightGBM 少数关键参数微调（p1 / p2 / p3 / p4）
**目的：** 在少量关键参数上微调，找更稳的树深、叶子数、树数配置。  
**结果：** 浅层稳健参数组 `p3_shallower` 表现最好或接近最好，更复杂的树不一定更优。

## 18. `lgbm_p3_shallower_core10_daily_top9` 候选主策略实验
**目的：** 验证浅层 + 核心 10 特征 + 每日 top9 是否能成为最优主线。  
**结果：** 这条主线在验证集和未触碰最终测试集上都表现很强，最终成为当前主策略。

## 19. `lgbm_p1_base_core12_daily_top9` 对照实验
**目的：** 比较另一个高表现候选在不同数据段上的泛化能力。  
**结果：** 它在 validation 上一度很强，但 final test 不如 `p3_shallower_core10`，泛化略差。

## 20. validation 冠军 vs final test 冠军核对
**目的：** 确认 validation 最优不一定是真正 final test 最优。  
**结果：** A/B 候选在 validation 与 final test 排名发生交换，说明最终必须以未触碰测试集结论为准。

## 21. 真实账户层回测口径确立
**目的：** 把研究从单笔收益推进到真实账户层资金曲线，明确初始资金、最大持仓数、单票上限。  
**结果：** 账户层结论和单笔层不完全一致，很多“单笔好”的策略到账户层未必最好。

## 22. 固定 10 仓等额分仓账户层回测
**目的：** 用简单、稳定、可解释的资金分配方式比较不同策略的真实账户层表现。  
**结果：** 这套等分仓位成为当前研究主口径，便于干净比较策略本身优劣。

## 23. 交易成本暂不纳入主口径讨论
**目的：** 先验证策略相对强弱，再考虑更真实实盘口径。  
**结果：** 当前主口径先不加手续费也能比较相对优劣，但后续接近实盘必须补成本与涨跌停约束。

## 24. 前置条件过强问题排查
**目的：** 检查为什么机器学习版和原始 `brick_filter.py` 结果过于接近。  
**结果：** 发现大概率是前置条件太强、候选池太窄、模型特征和原规则信息重叠太高。

## 25. 删除量能类前置条件实验设计
**目的：** 看放宽量能过滤后，模型是否能释放更多独立增益。  
**结果：** 候选池会明显变大，说明量能前置对样本空间约束很强。

## 26. 删除涨跌幅类前置条件实验设计
**目的：** 检查涨跌幅门槛是否过早限制了高质量样本进入模型排序阶段。  
**结果：** 候选池进一步扩大，表明涨跌幅前置也是强约束项。

## 27. 删除趋势类前置条件实验设计
**目的：** 检查趋势前置是否让模型只能在过窄的“好样本池”里微调排序。  
**结果：** 候选池暴增，说明趋势前置是非常强的样本压缩器。

## 28. relaxed BRICK 结构候选池实验
**目的：** 只保留 BRICK 形态本身，去掉量能/涨跌幅/趋势前置，观察宽松候选池下的研究空间。  
**结果：** 候选信号膨胀到 25 万级别，说明全删前置会让实验空间爆炸、计算量急剧上升。

## 29. research / validation / final_test 三段切分实验框架
**目的：** 避免把 validation 反复选优后还错称为 final test。  
**结果：** 研究流程变得更规范，也更能区分“候选领先”和“最终最优”。

## 30. stock signal cooldown 实验
**目的：** 降低同一股票短时间密集重复触发对统计的污染。  
**结果：** 冷却期有助于减少重复样本，使回测更接近真实交易频率。

## 31. relaxed 全量 5460 策略实验设计
**目的：** 覆盖模板构建、表示法、相似度算法、筛选方式、daily topN 的全组合搜索。  
**结果：** 组合空间极大，单次全量运行耗时可达数天，必须做缓存和加速优化。

## 32. 脚本性能优化（5460 → 156 基础配置）
**目的：** 把重复的模板构建、相似度打分、监督学习训练从完整策略级别降到基础配置级别缓存。  
**结果：** 运行逻辑显著优化，核心计算从 5460 次重算压缩到 156 次基础打分，再复用给筛选参数。

## 33. `pipeline_corr_dtw` 粗筛后精筛优化
**目的：** 降低 DTW 的超高计算开销。  
**结果：** 先用 corr 选 topK 模板再算 DTW 是正确方向，能明显减轻最慢 scorer 的负担。

## 34. `supervised_similarity` 缓存式训练思路
**目的：** 避免阈值/topN 改一下就重新构造特征和训练模型。  
**结果：** 正确做法是按基础配置只训练一次，后续阈值和 topN 全部复用 score。

## 35. `tqdm` 进度条加入实验
**目的：** 解决长任务看起来“卡住”但实际还在运行的问题。  
**结果：** 构建样本、基础打分、validation 评估都有可视化进度，更容易监控是否正常运行。

## 36. B1 / B2 / B3 / 单针 / BRICK 能否像 BRICK 一样做相似度 + ML 的讨论实验
**目的：** 评估其他策略是否也适合在原始规则候选池基础上做二次增强。  
**结果：** 结论是都能做，但应优先从 B2 开始，且更适合先做评分排序而不是先做复杂分类。

## 37. “只做评分排序，不做分类”方案讨论
**目的：** 降低五类策略增强的工程复杂度，先验证候选池上有没有二次排序空间。  
**结果：** 这条路线被认为更稳、更符合当前阶段，尤其适合 B2 / BRICK / B3。

## 38. B1 改成“两天确认版”思路讨论
**目的：** 把 B1 从“当天触发当天买”改成“昨天候选、今天温和阳线确认后买”，并与 B2 拉开。  
**结果：** 这种改法更稳，也更符合“低位修复确认”而不是“正式启动确认”的语义。

## 39. B1 与 B2 边界梳理实验
**目的：** 防止修改后 B1 和 B2 变成同一种买点。  
**结果：** 最终边界被明确成：B1 买修复确认，B2 买正式启动确认，不只是涨幅大小不同。

## 40. B3 逻辑审查实验
**目的：** 检查修改后的 B3 是否仍然保持“B2 后更稳的承接确认”语义。  
**结果：** 发现如果把前一天 B2 放得太宽、当天只要求缩量小阳，B3 会明显变松、失去“确认”味道。

## 41. B2 逻辑审查实验
**目的：** 检查修改后的 B2 是否还是真正的“正式启动阳线”。  
**结果：** 发现如果删掉最小涨幅、均量确认、实体要求，B2 会从“启动”退化成“普通修复阳线”。

## 42. 量化资金分配口径讨论实验
**目的：** 明确当前研究阶段该用等分还是加权仓位。  
**结果：** 结论是研究阶段先用等分做干净比较，实盘前再上轻度加权和更复杂组合优化。

## 43. 加权 vs 等分资金分配规则文档化
**目的：** 让 agent 在每次正式回测前先明确问用户要哪种资金分配方式。  
**结果：** 两套 md 规则定下后，回测前的假设会更清楚，不容易默认脑补。

# 当前阶段性总论

## 主策略结论
当前最可信的主策略是：

- `lgbm_p3_shallower_core10_daily_top9`

## 方法论结论
当前最成立的方法主线是：

- 原始候选池
- 相似度增强
- LightGBM 排序
- 账户层真实回测验证

## 研究流程结论
当前最稳的研究流程是：

- 先做规则候选池
- 再做评分排序
- 先用等分账户层口径验证
- 再逐步考虑更接近实盘的成本、涨跌停和轻度加权

## 风险提醒
当前仍需持续注意：

- validation 反复选优带来的过拟合风险
- relaxed 候选池过大带来的工程与计算爆炸
- B1 / B2 / B3 语义边界被改乱
- 候选池过窄导致模型增益有限
- 账户层最优与单笔最优混淆

## 44. B1 低位定义从 `J<13` 改为 `20日历史10%分位` 实验
**目的：** 检查 B1 的“低位”定义是否应该从绝对值阈值改成个股自适应历史分位。  
**结果：** 历史分位定义更符合不同股票的波动差异，后续也成为 B1 线斜率对比回测的基础口径。

## 45. B1 不同斜率过滤账户层对比实验
**目的：** 比较 `J<20日历史10%分位` 基础上，再叠加趋势线、多空线、均线不同窗口斜率大于 0 时，账户层哪个更优。  
**结果：** 在统一止盈止损和账户层口径下，最优的不是加任何额外斜率过滤，而是最基础的 `J低位 + 趋势线>多空线`。

## 46. B1 周线先筛实验
**目的：** 给 B1 增加周线背景过滤，要求周线趋势线大于多空线、周 MA20/30 三周斜率向上，并区分“周线强势”和“周线进碗”。  
**结果：** 周线条件可以作为 B1 的上层背景筛选，但 `周J低位` 与 `周MA60上行` 更适合作为附加标注而不是硬过滤。

## 47. B1 周线只保留完整周实验
**目的：** 避免未收盘周线引入未来信息，明确周线只能用周五收盘后形成的完整周 K。  
**结果：** 周线口径被固定为“只认完整周”，同时周均线斜率也明确为均线自身的变化而不是周收盘变化。

## 48. B1 “昨天候选、今天确认”两天版实验
**目的：** 把 B1 从“当天低位当天买”改成“昨天满足旧 B1，今天再用低位小阳确认后买”，进一步和 B2 拉开语义边界。  
**结果：** 这种改法更符合“低位修复确认”的交易语义，也避免了 B1 和 B2 都去抢同一类启动阳线。

## 49. B2 完美图成功案例上影线统计实验
**目的：** 从 `完美图/B2` 成功样本里反推 B2 上影线应该控制在什么范围内。  
**结果：** 成功 B2 中大多数 `上影线/实体` 落在 `1/3` 以内，`1/4` 偏严，`1/3` 是相对合理的硬门槛候选。

## 50. B2 点状出货 / 区间出货卖点回测实验
**目的：** 把“完美图/出货图”里总结的几类出货结构量化成 B2 的卖出标签，验证能否改善持有收益。  
**结果：** 当前这版出货标签对 B2 没有形成稳定增益，说明出货图逻辑有价值，但机器定义还不够准，暂时不适合直接并入 B2 正式卖出。

## 51. “击穿对手盘”概念引入与人工校准实验
**目的：** 区分“真实出货”和“击穿关键位置后的洗盘/试探”，避免把本不该卖的票误判成出货。  
**结果：** 击穿对手盘与出货点、区间出货被正式分开理解，但当前更适合做研究标签或卖出豁免候选，而不是直接自动执行。

## 52. “击穿对手盘”标签量化实验
**目的：** 把你口头定义的“跌破多空线/关键K后，次日快速拉回或缩量横盘再拉起”的结构写成可计算标签。  
**结果：** 第一版标签已经能落成代码，但误命中“没意义”和“出货”的样本仍偏多，说明这类结构更适合做动态状态机而不是静态标签。

## 53. B3 逻辑放宽与案例回放实验
**目的：** 检查当前 B3 是否能覆盖你主观认定的典型 B3 案例，并逐步放宽“前一日 B2”和“前前日 B1”的定义。  
**结果：** 把“前前日 B1”改成 `J<13 且前一日 J 更高` 后，你点名的宝丰能源、岱美股份、登云股份、东方盛虹这类 B3 案例都能被筛出来。

## 54. B3 上影线与振幅阈值放宽实验
**目的：** 检查 B3 的基础 B2 上影线阈值和 B3 当日振幅阈值是否过严。  
**结果：** 把上影线从 `1/4` 放宽到 `1/3`、把振幅从 `<4%` 放宽到 `<5%` 后，B3 的案例覆盖更符合主观样本，但真正决定能否入选的关键仍是链式结构本身。

## 55. 单针成功/失败样本对比实验
**目的：** 固定单针定义后，区分“为什么同样满足单针，有些成功有些失败”。  
**结果：** 成功与失败的主要分界不在“针本身”，而在趋势背景、前 10 日是否已涨、最近是否缩量回踩。

## 56. 单针 A/B/C 三分型重建实验
**目的：** 把单针从单一阈值扩展成“缩量回踩型 / 强趋势加速型 / 结构支撑型”三类，提高成功案例覆盖。  
**结果：** A+B+C 版本把可优化成功样本覆盖做到了 `15/15`，并且在当时的样本口径下没有引入新的失败案例。

## 57. 单针止盈止损 5 天 AB 实验
**目的：** 在固定 `T日信号、T+1开盘买、买入当日不能卖、最长持有5天` 的口径下，为单针找更均衡的退出方式。  
**结果：** 更均衡的卖法一度收敛到 `10%止盈 + 信号日最低价止损 + T+2起执行`，说明单针更适合短持有内的结构止盈，而不是长拿硬扛。

## 58. 单针推荐顺序与类型优先级实验
**目的：** 明确多类型单针同时命中时，哪一类优先级更高。  
**结果：** 当前排序固定为 `A型(缩量回踩) > B型(强趋势加速) > C型(结构支撑)`，其中纯 `A` 一般优于 `A+C` 这类复合信号。

## 59. BRICK 完美图重训练实验（去掉贴近趋势线）
**目的：** 在保留尽可能多成功案例的前提下，重新训练砖型硬条件，避免旧版过严。  
**结果：** 去掉“贴近趋势线”等过强前置后，真正还有信息量的约束主要收敛到 `趋势线>多空线`、`量比上限`、`涨幅上限`、`红砖质量` 这几类。

## 60. BRICK 三种砖色结构硬过滤实验
**目的：** 在不回到旧版 `3绿1红` 的前提下，加回更贴近主观砖型节奏的砖色约束。  
**结果：** “两绿一红 / 一绿一红一绿一红 / 两绿两红(前一日红很小)” 这三类结构能明显压缩噪声，同时对完美案例覆盖仍保持在较高水平。

## 61. BRICK 二值加减分排序实验
**目的：** 不再用连续型质量分，而是把成功共性和失败共性拆成清晰的加分项与扣分项，用于主流程排序。  
**结果：** 二值法更符合人工复盘语义，也更容易解释每只砖型为什么排前或排后。

## 62. BRICK 成功案例共性统计实验
**目的：** 从完美砖型案例里统计成交量、前几日回调、价格位置、N 型、连续绿砖、反包强度等共性。  
**结果：** 成功砖型更像“中高位活动区里，前 3 到 5 天先有压制，然后由一根明显不小、反包偏强的红砖把节奏扭转回来”，纯突破型反而不是主流。

## 63. BRICK 反例共性统计实验
**目的：** 对比失败或不想做的砖型案例，提炼哪些位置和结构容易出问题。  
**结果：** 反例常见问题不是“红砖不够大”，而是位置太靠近突破追价、位置太低像反抽、前面已有阶梯量或结构不干净。

## 64. BRICK `2倍量` 加分而非硬过滤实验
**目的：** 检查“信号日成交量 >= 前一天 2 倍”是否适合直接当砖型硬条件。  
**结果：** `2倍量` 对后续收益和上涨概率有加分作用，但会砍掉太多成功案例，因此更适合做额外标注或排序加分。

## 65. 风险标签公共模块建设实验
**目的：** 把“前期快速拉升”和“前期出货痕迹”统一量化成跨策略可复用的公共风险接口。  
**结果：** `market_risk_tags.py` 已经落成，第一版包含固定窗口涨幅、连续阳线段斜率、高位巨量阴线、假突破、顶部出货区、阶梯量阴跌等风险特征。

## 66. 风险标签在真实历史信号上的验证实验
**目的：** 不再依赖主观正反样本，而是直接用 `20260315/normal` 的历史数据验证：即使满足策略买点，触发某些风险标签后，后续是否明显更差。  
**结果：** 真实历史信号显示，“假突破/顶部出货痕迹”比简单的“几天涨太多”更稳定，而“连续阳线阶段斜率过陡”比固定窗口涨幅更接近你原先说的透支风险。

## 67. 风险标签硬过滤 / 扣分 AB 实验
**目的：** 检查 `B1 / B3 / 单针 / BRICK` 在加入风险标签后，哪些适合硬过滤，哪些只适合扣分提示。  
**结果：** 当前最像可落地方案的是：单针优先过滤“近20日高位巨量阴线”，B3 轻量过滤“顶部出货区”，BRICK 优先过滤“近20日假突破”或核心出货组合，而 B1 暂时更适合先打印风险提示。

## 68. `main.py` 慢点定位实验
**目的：** 找出主流程为什么显著慢于早期版本。  
**结果：** 真正的主因不是数据校验本身，而是多个策略重复计算同一批特征，以及 `brick_filter_case_pool` 这类额外全市场扫描。

## 69. `main.py` 共享缓存与并行优化实验
**目的：** 减少 `B1/B2/B3/单针/BRICK` 的重复读文件和重复算特征，并利用并行提升主流程速度。  
**结果：** 已经落地了基础特征缓存、周线缓存、主流程并行和 `brick_filter_case_pool` 拆出主流程，但仍需继续把更多重复计算收敛到统一底层。

## 70. `main.py` 去掉 B2 / BRICK+B2 主流程实验
**目的：** 在当前阶段只保留最需要的主流程策略，减少无效扫描和混乱输出。  
**结果：** 主流程已经缩减为 `B1 / B3 / 单针 / BRICK / 卖出监控 / LGBM_FINAL` 这类更聚焦的集合，B2 和 BRICK+B2 不再默认参与。

## 71. 公共接口梳理实验
**目的：** 明确哪些内容应该继续抽成公共接口，避免每个策略继续各写一套。  
**结果：** 已确认最值得继续抽象的层次包括：统一数据读取、基础特征、周线 bundle、结构标签、统一策略返回结构、统一回测执行接口。

# 补充阶段性总论

## 规则策略方向
当前规则策略的工作重点，已经从“继续靠拍阈值堆条件”逐步转向：

- 明确每个策略的语义边界
- 用主观案例校准结构定义
- 用真实历史信号验证哪些条件真的能减少后续下跌

## 风险过滤方向
当前对“绝对不做”这类风险的认识，已经从主观描述推进到可计算标签，最靠谱的主线不是：

- 简单几天涨太多

而更像：

- 近端假突破
- 顶部出货区痕迹
- 连续阳线段斜率过陡
- 高位巨量阴线

## 工程方向
当前工程优化的重点，已经不是继续堆策略数量，而是：

- 共享基础特征
- 共享周线结果
- 减少重复扫描
- 让主流程和研究脚本都能复用同一套底层接口
